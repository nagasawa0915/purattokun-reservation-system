## 🎯 エージェントの基本方針

### 💡 設計哲学: 理論より実践
- **完璧な設計 < 現実的な設計**: 中小規模プロジェクトで実際に使える設計を最優先
- **全機能実装 < 優先順位**: まず動くものを作り、段階的に改良
- **理想的アーキテクチャ < 保守しやすいコード**: チーム全員が理解できる範囲で設計

### 🚀 Quick Start 設計 3点セット

#### 1. **分離基準判定** (5分)
```
🔍 分離すべきかの即座判定:
□ 3つ以上のimport文がある
□ 1つの修正で2ファイル以上触る頻度が高い
□ ファイルサイズが300行を超えた
□ 同じコードを3箇所以上で書いている

✅ 2つ以上該当 → 分割検討
❌ 1つ以下 → そのまま継続
```

#### 2. **最小構成設計** (10分)
```
📁 最初に作るべきファイル構成:
core/           # 絶対に必要な機能のみ
├── main.js     # エントリーポイント
└── utils.js    # 共通関数

features/       # 個別機能（必要に応じて）
└── [機能名].js

config/         # 設定（設定が複雑な場合のみ）
└── settings.js
```

#### 3. **動作確認チェック** (5分)
```
✅ 最小チェックリスト:
□ メイン機能が動作する
□ エラーが出ない
□ 新機能追加時の修正箇所が明確
□ 他の開発者が理解できる構造
```

### 🎯 結合度判定チェック

#### 🔴 結合度が高い（要改善）
```javascript
// 悪い例: 密結合
class EditorPanel {
    save() {
        // 他クラスの内部状態を直接操作
        spineSystem.positions.x = this.xInput.value;
        spineSystem.positions.y = this.yInput.value;
        spineSystem.dirty = true;
        spineSystem.lastModified = Date.now();
    }
}
```

#### 🟢 結合度が低い（良い設計）
```javascript
// 良い例: 疎結合
class EditorPanel {
    save() {
        // インターフェース経由で操作
        const position = {
            x: this.xInput.value,
            y: this.yInput.value
        };
        this.spineSystem.updatePosition(position);
    }
}
```

## 🚨 よくある失敗パターンと対策

### ❌ 失敗パターン1: 過度な分割
```
問題: 10行のファイルを5個作成
対策: 最低50行以上or明確な独立機能のみ分割
```

### ❌ 失敗パターン2: 循環依存
```
問題: A.js → B.js → A.js の依存関係
対策: 共通機能をC.jsに抽出、A・BがCに依存
```

### ❌ 失敗パターン3: インターフェース複雑化
```
問題: モジュール間で10個の引数を渡す
対策: オブジェクト形式or設定オブジェクトで統一
```
